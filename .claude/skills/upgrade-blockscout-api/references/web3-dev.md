# Upgrade procedure — `web3-dev` target

Refresh the `web3-dev` skill's bundled OpenAPI specification and endpoint index from the latest commit of the upstream `pro-api` repository. The two artefacts the skill ships with are:

- `web3-dev/references/pro-api.json` — full Blockscout PRO API OpenAPI v3.0 document (~24,000 lines).
- `web3-dev/references/pro-api-index.md` — generated by-tag endpoint index derived from the JSON above.

The OpenAPI document itself is built from a git submodule, not authored in this repo. The pipeline is therefore: refresh the submodule → rebuild the OpenAPI doc inside it → copy the doc into the skill → regenerate the index.

## Prerequisites

- Python >= 3.9 (standard library only — the indexer has no third-party dependencies)
- `bash`, `jq`, and `oastools` available on `PATH` (used by the upstream `build-pro-api.sh` orchestrator and its child scripts)
- Network access to GitHub for the submodule fetch and to whatever sources `build-pro-api.sh` downloads (it caches commit-hash-keyed swagger files)
- Working directory is the repository root (parent of `web3-dev/` and `.memory_bank/`)

## Pipeline

Run every step in order. Each step depends on the output of the previous one. If a step fails, fix the cause before proceeding — do not skip ahead.

### Step 1 — Refresh the `pro-api` submodule

The OpenAPI source is the `pro-api` submodule at `.memory_bank/specs/web3-dev/references/pro-api`, which tracks the branch declared in `.gitmodules` (currently `feat/openapi-spec-intro`).

If the submodule directory is empty (first run after a fresh clone), initialise it:

```bash
git submodule update --init .memory_bank/specs/web3-dev/references/pro-api
```

To pick up the latest upstream commit on the tracked branch:

```bash
git submodule update --remote .memory_bank/specs/web3-dev/references/pro-api
```

After the update, confirm the submodule moved (and note the new commit) with:

```bash
git submodule status .memory_bank/specs/web3-dev/references/pro-api
```

If `git submodule status` reports the same commit it had before, the upstream branch hasn't moved and the rest of the pipeline is a no-op — you can stop here. Otherwise, continue.

### Step 2 — Rebuild `pro-api.json`

Run the upstream orchestrator that downloads the source swaggers, runs per-service extractions, and merges them into a single OpenAPI 3.0 document.

```bash
bash .memory_bank/specs/web3-dev/references/pro-api/openapi-spec/build-pro-api.sh
```

The script honours its own caching layer keyed by upstream commit hashes. Use `--force` if you need to re-download even when the cache is warm:

```bash
bash .memory_bank/specs/web3-dev/references/pro-api/openapi-spec/build-pro-api.sh --force
```

On success the script writes:

```
.memory_bank/specs/web3-dev/references/pro-api/openapi-spec/pro-api.json
```

Verify the file exists and is non-trivial in size before moving on:

```bash
ls -l .memory_bank/specs/web3-dev/references/pro-api/openapi-spec/pro-api.json
```

The script's exit codes are documented at the top of `build-pro-api.sh`: `0` success, `1` config error, `2` download failure, `3` extraction failure, `4` merge/validation failure. Any non-zero exit means the artefact is unusable — do not copy it forward.

### Step 3 — Copy the OpenAPI doc into the skill

Overwrite the bundled copy with the freshly built one:

```bash
cp .memory_bank/specs/web3-dev/references/pro-api/openapi-spec/pro-api.json \
   web3-dev/references/pro-api.json
```

This is a single file replacement — no merge, no patch. The bundled file is the authoritative copy from the agent's point of view; the submodule artefact only exists as a build-time intermediate.

### Step 4 — Regenerate the endpoint index

```bash
python3 .memory_bank/specs/web3-dev/tools/pro-api-indexer.py
```

The script reads `web3-dev/references/pro-api.json` (default input path) and **overwrites** `web3-dev/references/pro-api-index.md`. Output order is deterministic (tag sections sorted case-insensitively; entries within a section sorted by path then HTTP method), so a clean rerun on the same JSON produces a byte-identical index.

The indexer's exit codes: `0` success, `1` input file not found / unreadable, `2` input not valid JSON, `3` JSON has no `paths` key. Any non-zero exit means the index was not written.

## Verification

After completing all four steps, confirm:

1. `git submodule status .memory_bank/specs/web3-dev/references/pro-api` reports the expected new commit (and `git status` shows the submodule pointer change as a staged-or-unstaged repo modification).
2. `web3-dev/references/pro-api.json` is well-formed JSON containing a `paths` object:
   ```bash
   jq '.paths | length' web3-dev/references/pro-api.json
   ```
   should print a positive integer (number of operation paths).
3. `web3-dev/references/pro-api-index.md` exists, starts with `# PRO API Endpoint Index`, and has one `## <tag>` section per OpenAPI tag in the spec.
4. `git diff web3-dev/references/pro-api-index.md` shows additions/removals consistent with the spec changes (new endpoints appear under their tags; renamed/removed ones disappear). If the diff is empty after a non-trivial submodule bump, something silently failed in Step 2 or 3.
5. Re-running Step 4 against the same `pro-api.json` produces no further changes (the indexer is deterministic).
